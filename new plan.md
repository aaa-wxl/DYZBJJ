**当前已比较扎实的部分**
- 核心闭环：创建竞拍、启动、用户出价、排行榜、延时、封顶成交、取消、订单生成。
- Redis 实时路径：`RedisStore`、Lua 原子出价、幂等、Top 5、rank、取消/到期结算都已经有实现。
- 生产启动路径：`cmd/api/main.go` 已改成 Redis 优先，默认 Redis 不可用就失败。
- 多端同步：管理端、用户 A、用户 B 的 Playwright 测试已经覆盖主流程。
- 最近修复了“自己出价日志重复”的前端 bug，并加了 E2E 回归断言。

**我看到最该调整的地方**
1. **Redis 文档和实现有几处不一致**
   - 设计文档写的是每个竞拍独立 channel：`auction:{id}:events`。
   - 当前代码 `RedisBus` 用的是全局 channel：`auction:events`，靠事件里的 `AuctionID` 再广播到本地房间。
   - 设计文档写 `RedisStore.PlaceBid` 成功后发布事件；当前实际是 `AuctionService.broadcast` 通过 `EventPublisher` 发布。这其实更合理，因为它能在审计落库和 snapshot enrich 后统一广播，但文档要改。

2. **Redis 计划文档 checkbox 没同步**
   - `docs/superpowers/plans/2026-05-27-redis-auction-core.md` 里 Task 1-10 基本还都是未勾选。
   - 但代码实际已经完成大部分。
   - 这会影响你后面继续用 superpowers 时判断进度，也会影响项目材料的可信度。

3. **PRD 里的展示型功能还不够**
   - 用户端没有明显“直播间/固定视频”区域。
   - 缺少“领先 / 被超越 / 延时 / 成交”的情绪反馈动画。
   - 用户侧没有历史竞拍记录、模拟支付入口。
   - 管理端没有独立订单管理页，也不支持修改未开始竞拍规则。
   - 这些不是 Redis 核心，但对演示和评分很加分。

4. **稳定性表达还可以加强**
   - 当前 WebSocket 是手写握手和服务端推送，没有完整 ping/pong 心跳。
   - 前端有断线重连和 snapshot 恢复，这是好的。
   - 如果要贴合 PRD 的“WebSocket 稳定、心跳保活”，建议补一个轻量心跳或至少文档说明当前恢复机制。

5. **MySQL 目前只是预留**
   - Redis 设计文档明确说本阶段不迁 MySQL，这是合理的。
   - 但 PRD 评分里会看 MySQL/Redis 架构适配性。短期可以不迁移，但要在答辩材料中讲清楚：Redis 承接高频实时状态，文件仓储是演示持久化，MySQL schema 已预留，下一阶段迁移 repository。

**我建议的 3 个路线**
推荐路线 A：**先做 Redis 阶段收口**
- 更新 Redis 设计文档，让它和当前实现一致。
- 勾选/修正 Redis plan 进度。

- 缺点：短期不会增加可见功能。

路线 B：**优先补演示体验**
- 用户端加直播画面占位、领先/被超越提示、延时动画、成交结果入口。

- 优点：演示效果明显提升。
