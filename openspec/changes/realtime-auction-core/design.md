## Context

本 change 要实现直播竞拍系统的核心闭环：商家创建竞拍并配置规则，用户进入竞拍房间实时出价，系统在高并发下校验规则、更新竞拍状态、广播结果，并在成交后生成最小订单记录。

项目从空仓库开始，当前没有既有业务代码或规格。后端技术固定为 Go，前端使用 React + TypeScript。系统需要同时处理 RESTful API 的管理操作和 WebSocket 的实时事件，并使用 Redis 承担高频竞拍状态和并发控制，使用 MySQL 或 PostgreSQL 保存最终业务数据。

## Goals / Non-Goals

**Goals:**

- 建立 Go 后端分层架构，清晰隔离 API、WebSocket、领域服务、Redis 和数据库访问。
- 实现竞拍状态机，保证 `DRAFT`、`SCHEDULED`、`RUNNING`、`EXTENDED`、`SOLD`、`ENDED`、`CANCELLED` 的状态流转可控。
- 使用 Redis 原子操作处理高频出价，保证并发出价下的当前价、最高出价人、排行榜和结束状态一致。
- 使用 WebSocket 按 `auctionId` 管理房间并广播竞拍事件。
- 提供 React + TypeScript 的最小管理端和用户端，用于验证完整竞拍闭环。
- 成交后生成最小订单记录，形成从商品上架到成交的闭环。

**Non-Goals:**

- 不接入真实支付、退款或资金冻结。
- 不实现复杂 AI 模型调用。
- 不追求完整运营后台、权限体系和精细化数据报表。
- 不在本 change 中实现大规模可观测性平台。
- 不把精美动画作为核心验收目标，只保留必要的实时反馈。

## Decisions

### 后端采用 Go 分层单体

Go 后端先采用分层单体，而不是一开始拆微服务。建议目录按职责划分为 `cmd/api`、`internal/http`、`internal/ws`、`internal/domain/auction`、`internal/service`、`internal/repository`、`internal/redis`、`internal/config`。

理由：竞拍核心的一致性边界集中在竞拍领域内，分层单体更容易保证事务、状态机和 WebSocket 广播的一致协作。微服务会提前引入服务发现、分布式事务和链路追踪成本，不适合第一个核心闭环。

备选方案是 Node.js 或微服务架构。Node.js 符合课题允许范围，但用户已明确后端固定使用 Go；微服务可作为后续高并发扩展方向，不作为首个 change 的基础。

### Redis 承担竞拍热点状态和原子出价

每个竞拍在 Redis 中维护热点状态，例如当前价、最高出价人、结束时间、状态、排行榜和已处理的 `requestId`。出价请求通过 Lua 脚本或 Redis 事务完成原子校验和更新。

理由：出价是最高频路径，直接用数据库行锁会增加延迟和锁竞争。Redis 原子脚本可以把状态校验、价格更新、排行榜更新、幂等检查和自动延时判断放在一次操作中完成。

备选方案是数据库事务加行锁。它实现简单但在 100+ 并发出价和实时刷新场景下更容易形成瓶颈，可作为 Redis 不可用时的降级兜底，不作为主路径。

### 数据库保存权威业务记录

数据库保存商品、竞拍配置、竞拍最终状态、出价流水和订单记录。Redis 中的热点状态用于实时路径，关键状态变化需要异步或同步落库，竞拍结束时以数据库记录作为最终审计来源。

理由：Redis 适合实时状态，但不适合作为长期审计账本。出价流水和订单必须可追溯，数据库更适合作为最终一致的业务记录。

### WebSocket 使用房间级 Hub

Go 后端维护 WebSocket Hub，以 `auctionId` 作为 room key。用户进入竞拍房间后订阅对应 room，出价成功、倒计时延时、排名变化、竞拍结束等事件只广播给同一 room。

理由：房间隔离可以避免不同直播间互相干扰，也方便后续做单房间压测、水平扩展和 Redis Pub/Sub 跨实例广播。

首个 change 可先实现单实例 Hub；如果需要多实例部署，再引入 Redis Pub/Sub 或 Stream 做跨进程广播。

### 状态机集中在领域服务

竞拍状态流转必须由竞拍领域服务统一处理，REST API、WebSocket handler 和定时任务都不能绕过状态机直接改状态。

理由：竞拍规则涉及启动、延时、封顶成交、自然结束和异常取消。如果状态写入分散，容易出现重复成交、已取消后仍可出价、结束时间不一致等问题。

### 前端先做最小双端验证

React + TypeScript 前端先实现两个最小入口：商家管理页和用户竞拍页。商家页支持创建竞拍、启动竞拍、取消竞拍、查看成交结果；用户页支持进入房间、查看当前价和倒计时、提交出价、接收实时事件。

理由：首个 change 的目标是验证闭环和技术核心，不应把大量时间投入完整运营后台和视觉动画。

## Risks / Trade-offs

- Redis 和数据库状态不一致 → 出价成功后记录出价流水，竞拍结束时执行收敛流程，把 Redis 最终状态写入数据库，并通过补偿任务检查未落库记录。
- WebSocket 单实例 Hub 无法覆盖多实例部署 → 首个 change 明确以本地单实例为目标，接口和事件结构预留 Redis Pub/Sub 扩展点。
- 客户端倒计时与服务端时间偏差 → WebSocket 事件中发送服务端 `serverTime` 和 `endsAt`，前端根据服务端时间修正本地倒计时。
- 高频出价导致重复请求或重复扣减 → 每次出价必须携带 `requestId`，Redis 原子脚本检查幂等键，重复请求返回同一处理结果或明确拒绝。
- 封顶价和自动延时同时触发 → 状态机规定封顶价优先级高于自动延时，达到封顶价后直接进入 `SOLD`。
- 出价广播失败但状态已更新 → 出价 API 返回最新状态，WebSocket 支持重连后拉取快照，避免完全依赖单次广播。

## Migration Plan

这是新项目的首个核心 change，不涉及既有数据迁移。

实施顺序：

1. 创建前后端工程骨架和本地配置。
2. 建立数据库表和 Redis key 约定。
3. 实现竞拍领域模型、状态机和出价原子校验。
4. 实现 RESTful API 和 WebSocket Hub。
5. 实现最小 React 管理端和用户端。
6. 添加核心测试、手工验证脚本和本地运行说明。

回滚策略：由于没有线上旧版本，回滚以删除本地数据、清空 Redis key、回退代码 change 为主。

## Open Questions

- 数据库最终选择 MySQL 还是 PostgreSQL，需要在实现前根据本地环境确定。
- WebSocket 库选择 `nhooyr.io/websocket` 还是 `gorilla/websocket`，实现时应优先选择维护状态和项目适配度更好的方案。
- 是否需要在首个 change 中加入简单压测脚本，还是放入后续 `high-concurrency-polish` change。
